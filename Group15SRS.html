Part 1: Introduction

1.1        Purpose

The purpose of this document is to present a detailed description of our Geospatial Architecture Database. It will list and detail the functional and nonfunctional requirements in relation to the functionality, interface and constraints of the system.

1.2        Scope

This document will provide information about the Database, including the functions:

  1. Display weather patterns for User’s requested position.
  2. Display real time weather for requested location.
  3. Provide construction advice based on local weather/weather patterns.
  4. Allow users to export data from the Database.

1.3        Intended Audience

The intended Audience of this SRS is for both users and developers of the Geospatial Architecture Database project at any and all stages of development.

1.4        Document Conventions

This document uses an IEEE outline format to separate information into easily accessible sections as well as help with readability and the comprehension of the Geospatial Architecture Database system.

1.5        Definitions

Term

Definition

GAD

Acronym for Geospatial Architecture Database

User

The user of the Database

Leaflet + OpenStreetMap

Open-source JavaScript library (Leaflet) for displaying interactive maps; tile data and geocoding (Nominatim API) provided by the OpenStreetMap project.

Part 2: Overall Description

2.1        Product Perspective

The GAD is a database made to inform the user of weather conditions/climate as well as advice surrounding current or future construction projects. The GAD will allow users to request a potential or current build site, and the system will provide the weather and safety precautions surrounding that location to the said user. For example, if a city such as Tampa was selected, the GAD would present the weather history including the strong pattern of hurricanes and advise the user to build a strong foundation accordingly.

2.2        Product Functions

2.2.1        Location Input

GAD will allow the user to input any valid location.

2.2.2        Leaflet + OpenStreetMap integration

GAD will have Leaflet + OpenStreetMap integration, providing a clean, open-source visual interface for selecting and viewing locations.

2.2.3        Weather History

Upon inputting a location, GAD will provide an accurate history of weather conditions and disasters for said location.

2.2.4        Construction Safety Calculations

After retrieving weather data, GAD will calculate safety recommendations depending on the given data.

2.2.5        Data Export

If requested, GAD will allow users to export given data in relation to their chosen location including weather metrics and construction advice.

2.3        User Interface/Characteristics

This system is designed for users familiar in and around architecture to interpret the results, as well as basic knowledge on how to use a computer. Users will be able to view their inputted location on a map, with a brief visualization of harsh weather, as well as export the given data.

2.4        End-User Operation Environment

Users will be able to access the database through the internet on their preferred browser, they will be able to save files to their local disk.

2.5        Design and Implementation Constraints

The system will be designed using the Python coding language thus anyone wishing to modify or update the system will be required to know python. A basic internet connection will also be required to access the Database.

Part 3: Functional Requirements

Summary

The website will display weather history data and construction tips based on a location specified by the user using Leaflet with OpenStreetMap tiles as the base using either an address or GPS coordinates.

The Database will use the following tools:

  • Github to store the repository of the code.
  • National Weather Service API to access weather related data from the National Weather Service.
  • Python for primary coding language.
  • Leaflet (open-source JavaScript library) for interactive map rendering.
  • OpenStreetMap Nominatim API for address and coordinate geocoding.
  • Various AI software to assist in coding.

3.1        Leaflet + OpenStreetMap Integration

Accesses inputted GPS coordinates and addresses and displays those same coordinates/addresses in an interactive Leaflet + OpenStreetMap environment for readability.

3.1.1        Use Case 1: GAD displays requested location.

Brief Description:

  • The system accesses the OpenStreetMap Nominatim API to display an inputted location to the user.

Pre-Conditions:

  • Active Internet Connection, current access to the GAD website, User prompted to input address or coordinate.

                Post-Conditions:

  • The user is presented with a visual representation of their requested location.

                Events Flow:

  1. User accesses GAD.
  2. User prompted to input address or coordinate.
  3. User inputs address or coordinate using their keyboard.
  4. System calls the OpenStreetMap Nominatim API for the same location.
  5. The user is displayed a visual representation of their inputted location.

                Alternate Flow and Exceptions:

  • If inaccurate or unavailable address or coordinate is received, or if there is no active internet connection. An error message will instead be displayed.

3.1.2        Use Case 2: GAD cannot call Location

Brief Description:

  • The system accesses the OpenStreetMap Nominatim API to display an inputted location to the user, but cannot find the location.

Pre-Conditions:

  • Active Internet Connection, access to the GAD website, user prompted to enter address/coordinate.

                Post-Conditions:

  • The user is presented with an error stating “Cannot find requested location, please try again.”

                Events Flow:

  1. User accesses GAD.
  2. User prompted to input address or coordinate.
  3. User inputs address or coordinate using their keyboard.
  4. System calls the OpenStreetMap Nominatim API for the same location.
  5. OpenStreetMap Nominatim is unable to return the location.
  6. System displays an error message.

                Alternate Flow and Exceptions:

  • If an accurate and available address or coordinate is received, the user will be displayed their inputted address. Or if there is no internet connection, a separate error will be displayed.

3.2        Weather Data Retrieval/Analysis

The system will retrieve historical weather data. From the National Weather Database and analyse the data for dangerous weather history or current unsafe conditions.

3.2.1        Use Case 1: GAD Accesses Weather Data.

Brief Description 

  • The system accesses the API for the National Weather Database to retrieve data to relay to the user.

Pre-Conditions

  • Accurate address or coordinate inputted to the system and displayed, Active internet connection.

Post-Conditions

  • The user is displayed weather data related to their currently inputted location.

                Events Flow:

1: User accesses website.

2: User inputs desired location.

3: Weather Data relevant to the desired location is displayed.

Alternate Flow and Exceptions

  • If inaccurate or unavailable address or coordinate is received, or if there is no active internet connection. An error message will instead be displayed.

3.3        Data Export

Users will be allowed to export desired weather data in the format of PDF/CSV.

3.3.1        Use Case 1: Construction recommendations export

                Brief Description

  • Users will export the construction recommendations suggested in PDF/CSV.

                Pre-Conditions 

  • System has analyzed information and generated construction recommendations.
  • User has selected export option  

                Post-Conditions 

  • File is successfully generated and saved onto the user's local device.

                Events Flow:

  1. User selects the “Export Construction Recommendations” option
  2. The system prompts user to select format (PDF/CSV)
  3. User selects the format
  4. System compiles and generates the downloadable file
  5. Download is saved onto the users local device
  6. System displays confirmation message

                Alternate Flow and Exceptions

  • The system failed to download file -> Displays error message to screen
  • Storage permission denied -> system alerts the user

 

3.3.2        Use Case 2: analyzed weather history export

Brief Description 

  • The user exports the analyzed weather history for location

                Pre-Conditions

  • The weather analysis has been completed

                Post-Conditions

  •  The export has been saved to the users local device

                Events Flow 

  1. User selected the “Export Weather history” option
  2. System ask the user what format they would like
  3. User selects format
  4. System compiles and generates the downloadable file
  5. Download is saved onto the users local device
  6. System displays confirmation message

                Alternate Flow and Exceptions -

  • If the data is unavailable -> Export option is disabled
  • If the file generation fails -> User will be displayed Error message with reason.

3.4        Assessments of Environment Constraints and Construction Data

The system should generate a risk assessment score for selected locations and generate a list of construction recommendations based on analyzed weather data.

3.4.1        Use Case 1:

                Brief Description:

  • System analyzes weather data, generates risk assessments.

                Pre-Conditions

  •  Weather data is successfully retrieved.

                Post-Conditions

  • A risk assessment with score and list of recommendations.

                Events Flow 

  1. Receives data. Evaluates frequency and severity of severe weather.
  2.  System assigns weighted values to each risk category.
  3. System then gives construction recommendations based on the risk score.

                Alternate Flow and Exceptions:

  • Insufficient data exist -> System notifies user that assessment may be limited 

3.5        Error Handling

The system should handle Errors that occur throughout the user experience.

3.5.1        Use Case 1: Failure to find Address

                Brief Description

  • User inputs coordinates and the system cannot resolve to a location.

                Pre-Conditions 

  • User has entered coordinates.

                Post-Conditions 

  • An error message is displayed. User is then returned to the search field.

                Events Flow:

  1. User inputs coordinates.
  2. System sends input to OpenStreetMap Nominatim.
  3. OpenStreetMap Nominatim returns no matching location.
  4. System then returns an error message, “Location not found”
  5. Prompt user to re-enter location.

                Alternate Flow and Exceptions

  • If repeated errors occur -> System may suggest the user to input GPS coordinates instead.

3.5.2        Use Case 2: No internet

                Brief Description

  • The system detects that there is no internet connection.

                Pre-Conditions

  • User attempts to access the weather database.

                Post-Conditions

  • User is notified that internet connectivity is required.

                Events Flow

  1. System attempts to access API
  2. Connection fails
  3. System detects connection failure
  4. System displays no connection message to user
  5. System pauses any data retrieval attempts.

                Alternate Flow and Exceptions

  • System connection to internet is restored -> System allows user to retry previous actions/request

Part 4: Nonfunctional Requirements

4.1        Reliability

The Database should not take longer than 5 seconds to locate the user’s requested location and pull data from the National Weather Service.

4.2        Robustness

The System will display any errors to the user that were run into. Such as a lack of internet connection, failure to locate a requested address or coordinate or similarly if that said address or coordinate appears to not be a real location for construction like an ocean.

4.3        Maintainability

Modifications shall be made periodically to upkeep the web based system as well as update any bugs or errors found during the system’s availability. It will also allow any necessary developers access to modify it themselves.

4.4        Security

System will not store personal information. Information transfer over HTTPS.

4.5        Usability

Accessibility features, clean user interface, easy to read data for a broad audience, to reflect the basic knowledge of computers needed to use the program.
