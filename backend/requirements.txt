# ─── Web framework ───────────────────────────────────────────────────────────
# Satisfies:
#   §2.4 End-User Operation Environment — "access the database through the
#        internet on their preferred browser"
#   §2.5 Design and Implementation Constraints — Python primary language
#   §3.1 Leaflet + OpenStreetMap Integration — serves the frontend that
#        hosts the Leaflet map
#   §4.1 Reliability — lightweight WSGI app keeps response time under the
#        5-second budget
Flask==3.0.0

# ─── HTTP client for external APIs ───────────────────────────────────────────
# Satisfies:
#   §3.1 Leaflet + OpenStreetMap Integration — calls the OpenStreetMap
#        Nominatim API (https://nominatim.openstreetmap.org) to resolve
#        addresses/coordinates for display on the Leaflet map
#   §3.2 Weather Data Retrieval/Analysis — calls api.weather.gov (NWS)
#   §3.4 Assessments of Environment Constraints — fetches active alerts
#        feeding the risk-score pipeline
#   §3.5 Error Handling — RequestException handling enables the
#        "no internet" / "service unavailable" flows
requests==2.31.0

# ─── PDF report generation ───────────────────────────────────────────────────
# Satisfies:
#   §3.3 Data Export — "Users will be allowed to export desired weather
#        data in the format of PDF/CSV"
#   §3.3.1 Use Case 1 — Construction recommendations export
#   §3.3.2 Use Case 2 — Analyzed weather history export
# (CSV export in §3.3 is handled by the Python standard library `csv`
#  module on the frontend side and requires no extra dependency.)
reportlab==4.0.7

# ─── Database / ORM ─────────────────────────────────────────────────────────
# Satisfies:
#   §4.3 Maintainability — Reference data (state hazard profiles, IECC zones,
#        building codes, historical events, decadal trends, risk categories,
#        construction tips) lives in a SQLite database accessed via the
#        SQLAlchemy ORM, replacing the previous Python-dict approach. This
#        gives a single source of truth for all reference data, lets us
#        version the schema with Alembic in a follow-up, and provides a
#        natural extension point for user accounts and saved sites.
# Storage backend: SQLite (file at backend/gad.db by default; in-memory
# for tests; override with GAD_DATABASE_URL env var). Python stdlib `sqlite3`
# bindings are used; no additional driver pin is required.
SQLAlchemy==2.0.36

# ─── Schema migrations ──────────────────────────────────────────────────────
# Satisfies:
#   §4.3 Maintainability — Alembic versions the database schema. init_db()
#        calls `alembic.command.upgrade(cfg, "head")` on every boot so a
#        fresh checkout is always at the latest revision; the seed loader
#        runs after migrations succeed.
# Schema-change workflow:
#   1. Edit backend/db/models.py
#   2. `alembic revision --autogenerate -m "describe change"`
#   3. Hand-review the generated file in alembic/versions/
#   4. Commit. Next app boot applies the migration automatically.
alembic==1.13.3
